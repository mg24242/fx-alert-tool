# FX運用ナレッジ v2

## 1. 運用の目的と前提

このドキュメントは、本体ナレッジ「朝のルーティン運用ナレッジ」のFX領域を切り出した補助運用ルールである。本体ナレッジは日本株デイトレを主軸とする設計のため、FX固有の知識（特に介入警戒モードと要人発言の解釈）を独立させた。

- **運用上の位置づけ**：FXは本体ナレッジの「地合いに応じた切替先」として参戦する選択肢。日本株デイトレが取れない時間帯（15:30以降）または日本株の地合いが弱い日に、ドル円・ユーロドルでトレードを行う。
- **本ナレッジの主目的**：(1) 介入警戒モードの判定基準を言語化すること、(2) 要人発言Lvを自動分類できる仕様を整備すること、(3) Lv判定に基づくエントリー方向制限を明文化すること、(4) 要人発言監視ツールの実装可能な仕様書を提供すること。
- **ナレッジ分離の根拠**（2026/04/30実運用2日目に判明）：4/30の口先介入（片山財務相・三村財務官）でドル円が3円急落した実体験を踏まえ、介入警戒モードの設計が必要となった。本体ナレッジに統合すると主軸（株デイトレ）がぼやけるため、FX専用の補助ナレッジとして分離。
- **本体ナレッジとの連携**：本体4-6（通知後の判断フロー）でFX参戦が選択された場合、本ナレッジのLv判定と介入警戒モードを参照する。

## 2. 介入警戒モードの定義

### 2-1. モードの存在意義

通常時、ラウンドナンバーや日足レベルの床は「機関投資家の指値が集中する反発ポイント」として機能する。だからこそ押し目買い（逆張り）が成立する。

しかし以下の条件が揃った日は、同じ床が「ストップロスが集中するポイント」に変質する。

- USD/JPYが直近1年高値圏 or 5円刻みの心理的節目に接近している
- 日本の財務省・財務官による口先介入Lv3以上の発言が出ている
- 流動性が薄い（GW・年末年始・米祝日など）

この状態では、サポートを下抜けた瞬間にストップ狩りが連鎖し、通常時のATRの5〜10倍の値幅が出る。これを介入警戒モードとして識別し、エントリー方向を制限する。

### 2-2. 発動条件（AND条件）

以下の2つを両方満たした時、介入警戒モード発動。

- **価格条件**：USD/JPYが直近1年高値から3円以内、または5円刻みの節目（155, 160, 165...）の上1円以内
- **発言条件**：当日中に発言Lv3以上が観測されている（3章のLv判定表参照）

### 2-3. 発動時の制限

| 制限項目 | 内容 |
|---------|------|
| 押し目買い（逆張り） | **完全封印** |
| 新規エントリー方向 | ショートのみ可、ロングは終日封印 |
| ロット | 通常の半分に縮小 |
| 既存ロング | 即時利確 or 損切りラインを浅め（通常の半分）に再設定 |
| 既存ショート | 保有継続可、3円下を超えたら段階利確 |
| ノーポジ判断 | Lv4が観測されたら原則ノーポジ |

### 2-4. モード解除条件

- 当日のNY市場クローズ（翌6:00）まで継続
- もしくは、財務相・財務官から明確な「警戒緩和」発言が出た場合（過去事例ではほぼない）

## 3. 発言Lv判定表

### 3-1. Lv判定の基本方針

財務省・財務官の発言には**業界で共有された序列**がある。同じ「断固たる」でも、定例会見の枕詞か、緊急ぶら下がりかで重みが違うため、最終判定は人間（またはAI）の文脈読解が必要だが、自動分類のためのキーワード辞書を整備する。

**発言の方向性は一方通行**：日本の通貨当局（財務省・財務官・日銀総裁）の発言は構造上ほぼ「円安けん制」のみ。発言Lvが上がれば、方向性は**円高方向で確定**する。

**例外**：米要人（パウエル、ベッセント財務長官等）の発言は両方向に振れうるため、Lv判定とは別に文脈解釈が必要。

### 3-2. Lv判定キーワード辞書

| Lv | 名称 | 典型表現（日本語） | 意味 | 想定変動幅（数時間〜1日） |
|----|------|-------------------|------|--------------------|
| Lv1 | 通常 | 「為替は安定的が望ましい」「市場をよく注視している」 | 定例コメント | 反応軽微（0.1〜0.3円） |
| Lv2 | 観察強化 | 「過度な変動」「投機的な動き」「憂慮している」「ファンダメンタルズに沿わない」 | 警戒姿勢 | 0.3〜0.7円の下押し |
| Lv3 | 警戒 | 「フリーハンドがある」「あらゆる手段を排除しない」「準備はできている」「タイミングを見ている」 | 介入準備示唆 | 0.7〜1.5円の調整 |
| Lv4 | 最終警告 | 「断固たる措置」「いつでも対応できる」「最後の退避勧告」「行動するときが来た」 | 介入直前 | **1.5〜3円の急落**（薄商いだとさらに拡大） |
| Lv5 | 実弾介入 | （発言ではなく実際の介入） | 実介入実施 | 3〜5円の急落、その後ほぼ戻らず |

### 3-3. 自動Lv判定のルール

ツール（要人発言監視ツール、6章参照）でLvを機械判定する際は、以下の優先順位で適用する。

1. **タイトルマッチ優先**：ニュース見出しに含まれるキーワードを最優先（重要発言は見出しに入る）
2. **強いキーワードが優先**：Lv4キーワードとLv2キーワードが同時にマッチしたらLv4と判定
3. **人物マッチ必須**：監視対象人物（5章参照）の発言として明示されているもののみ対象
4. **過去発言の引用は除外**：「過去に〜と発言した」「以前の〜」を含む文脈は判定対象外（誤検知防止）

### 3-4. 過去事例（実Lv判定の参考）

- **2026/04/30（実運用2日目）**：片山財務相「断固たる措置をとるタイミングが近づいている」→ Lv4。三村財務官「最後の退避勧告だ」→ Lv4。実測変動幅3円（薄商い増幅あり）
- **2024/07**：神田財務官「断固たる措置」連発 → Lv4 → 7/11に実介入Lv5、約5円の急落
- **2024/04/29**：実介入Lv5、約4円の急落
- **2022/09/22**：実介入Lv5、約5円の急落

### 3-5. 割り切りの根拠

- 完全自動化は難しい。発言は文脈依存（同じ「断固たる」でも、定例会見の枕詞か緊急ぶら下がりかで重みが違う）
- ただしLv1〜5の序列を頭に入れておけば、ニュースを見た瞬間の反応速度が劇的に変わる
- ツールはLv判定の一次フィルタに徹し、最終判定は人間（またはAIへの判定依頼）に委ねる
- **ツールの役割は「気づきを与えること」のみ**。精度100%は目指さない。誤検知（False Positive）はノイズとして許容、検知漏れ（False Negative）の方が痛いので、辞書はやや広めに設定する

## 4. Lv別エントリー制限ルール

### 4-1. Lv2発言時

- 既存ポジション：影響軽微、保有継続
- 新規エントリー：通常通り、ただし高値圏での押し目買いはロット2/3に縮小
- 監視強化：当日中に追加発言が出ないかニュース監視

### 4-2. Lv3発言時

- 既存ロング：ロット半減 or 早期利確
- 既存ショート：保有継続可
- 新規エントリー：**ロング封印**、ショートは可（浅めの利確、0.7〜1.5円目処）
- 介入警戒モード発動条件（2-2）の片方を満たした状態。価格条件も満たせばモード発動

### 4-3. Lv4発言時

- 既存ロング：**即時撤退**（戻りを待たない、損切りラインまで耐えない）
- 既存ショート：保有継続だが、3円下を超えたら段階利確
- 新規エントリー：**ショート方向のみ可**、ロングは終日封印
- 価格目標：直近高値から1.5〜3円下を一旦の目処として認識
- ノーポジが原則。エントリーするなら少額・短期

### 4-4. Lv5（実介入）発生時

- 押し目買い完全封印（介入後は数日〜数週間、戻り売り優位の地合いが続くのが過去パターン）
- 既存ロングは即時撤退
- 新規エントリーはショートのみ、それも翌日以降の戻りを待つ
- 介入実施後は数時間ボラティリティが極めて高い。落ち着くまで様子見

## 6. 要人発言監視ツールの仕様書

**この章はCursorに渡して実装させる前提で書かれている。実装担当者（Cursor）はこの章を上から順に読めば、設定・実装・テスト・デプロイまで完遂できる。**

### 6-1. ツールの目的と非目的

**目的**
- 要人発言を自動検知し、Lv判定してDiscordに通知する
- 通知疲れを起こさない設計（多層フィルタとクールダウン）
- ノイズフィルター精度を運用しながら育てる（過去ログを残す設計）

**非目的（やらないこと）**
- 完全自動売買への接続（あくまで気づきを与えるだけ）
- リアルタイム秒単位の検知（5分遅延で十分）
- 100%の精度（誤検知も検知漏れも許容、運用しながら辞書を改善）

### 6-2. システム構成

```
[情報源（RSS）]
 ├─ ロイター日本
 ├─ ブルームバーグ日本
 ├─ 日経速報
 └─ Yahoo!ニュース 経済（補完）
　　　↓ 5分ごとにポーリング（GitHub Actionsのcron）
[フィルタ層]
 ├─ レイヤ0：重複排除（前回までに通知済みかチェック）
 ├─ レイヤ1：必須キーワード（為替/ドル円/介入/円安/円高 のいずれか）
 ├─ レイヤ2：人物マッチ（5章のリスト、いずれかが含まれる）
 ├─ レイヤ3：強度判定（3-2のキーワード辞書でLv2〜5を分類）
 ├─ レイヤ4：時間帯フィルタ
 └─ レイヤ5：クールダウン（同一人物の発言は1時間以内は再通知しない）
　　　↓
[通知層（Discord Webhook）]
 ├─ Lv2 → 低優先チャンネル（ログのみ、プッシュ通知OFF推奨）
 ├─ Lv3 → 中優先チャンネル（プッシュ通知ON）
 └─ Lv4-5 → 高優先チャンネル（プッシュ通知ON）
　　　↓
[ステート保存層]
 通知済み記事ID・最終通知時刻をJSONファイルで保持（GitHub Actionsの場合はリポジトリにコミットして引き継ぐ）
```

### 6-8. Lv判定ロジックの実装方針

```python
def judge_lv(title: str, content: str, keywords: dict) -> int:
    """
    記事のタイトルと本文からLvを判定する。
    タイトルマッチを優先（重要発言は見出しに入るため）。
    強いキーワードが優先（Lv4 > Lv3 > Lv2）。
    """
    # タイトル優先で判定
    for lv in [4, 3, 2]:
        for kw in keywords[f"lv{lv}"]:
            if kw in title:
                return lv

    # タイトルになければ本文を見る
    for lv in [4, 3, 2]:
        for kw in keywords[f"lv{lv}"]:
            if kw in content:
                return lv

    # どれにもマッチしなければLv1（通常）
    return 1
```

**精度に関する割り切り**：
- 単純なキーワードマッチで十分。NLPや機械学習は使わない
- 誤検知（False Positive）は許容。Lv2を多めに拾う方が、Lv4を見逃すよりマシ
- 検知漏れ（False Negative）の方が痛いので、辞書はやや広めに

### 6-9. エラーハンドリング・リトライ方針

| エラー | 挙動 |
|--------|------|
| RSS取得失敗（HTTPエラー、タイムアウト） | 当該フィードのみスキップ、ログに警告。次回実行時に再試行 |
| RSSパースエラー（不正なXML） | 当該フィードのみスキップ、ログに警告 |
| Discord Webhook送信失敗 | 1回だけリトライ（5秒後）、それでも失敗ならログに記録、処理続行 |
| 設定ファイル読込失敗 | プロセス即時終了（修正必須） |
| ステートファイル読込失敗 | 空の状態で開始（過去通知履歴は失われるが、ツールは動き続ける） |

**タイムアウト設定**：
- RSS取得：10秒
- Discord通知：5秒

**リトライしない理由**：5分後に再実行されるため、致命的でないエラーは次回に任せる方がシンプル。

### 6-11. GitHub Actions ワークフロー仕様

`.github/workflows/monitor.yml`：

```yaml
name: FX Alert Monitor

on:
  schedule:
    - cron: '*/5 * * * *'  # 5分ごと
  workflow_dispatch:        # 手動実行も可能

jobs:
  monitor:
    runs-on: ubuntu-latest
    permissions:
      contents: write       # state/notified.jsonをコミットするため
    steps:
      - uses: actions/checkout@v4

      - name: Setup Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: Install dependencies
        run: pip install -r requirements.txt

      - name: Run monitor
        env:
          DISCORD_WEBHOOK_LOW: ${{ secrets.DISCORD_WEBHOOK_LOW }}
          DISCORD_WEBHOOK_MID: ${{ secrets.DISCORD_WEBHOOK_MID }}
          DISCORD_WEBHOOK_HIGH: ${{ secrets.DISCORD_WEBHOOK_HIGH }}
        run: python -m src.main

      - name: Commit state
        run: |
          git config user.name "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add state/notified.json
          git diff --staged --quiet || git commit -m "Update state [skip ci]"
          git push
```

**注意点**：
- `[skip ci]`をコミットメッセージに入れることで、ステート更新コミット自体がワークフローを再起動しないようにする
- GitHub Actionsの無料枠は月2000分。5分ごと実行で月8640分必要なため、**パブリックリポジトリ推奨**（無料無制限）。プライベートにしたい場合は10分ごと実行に変更（月4320分でも超えるため、月2回に1回スキップする等の調整が必要）

## 7. AIへの委譲フロー

ツールが通知を出したあと、より深い判定が必要な場合はAIに委譲する。

### 7-1. 委譲コマンド「警戒Lvチェック」

ユーザーは通知を見て「これは判断に迷う」と感じたら、AIに以下のコマンドを投げる。

```
警戒Lvチェック：[発言内容 or ニュースURL]
```

AIは以下を返す。

1. 発言の最新文脈をweb検索で確認
2. Lv判定（自動判定の補正）
3. 直近の市場反応（既に何円動いたか）
4. 推奨アクション（既存ポジション・新規エントリーへの影響）

### 7-2. 委譲コマンド「介入警戒モード判定」

価格と発言の両条件を踏まえて、現在介入警戒モードに入っているかを判定する。

```
介入警戒モード判定お願い
```

AIは以下を返す。

1. 現時点のUSD/JPY価格と直近1年高値・節目との位置関係
2. 当日の発言Lv観測状況
3. モード発動状況（発動中 / 発動条件1つだけ満たす / 通常）
4. 推奨アクション

### 7-3. 朝のレポートへの統合

本体ナレッジの「朝の地合い確認レポート」（本体3章）の【最重要】変化要因欄で、当日の介入警戒レベルを明記する。

例：
- 「介入警戒Lv2：高値圏だが発言は通常範囲」
- 「介入警戒Lv4観測中：昨夜片山財務相の最終警告。本日FXロング全面封印推奨」

これによりユーザーは寄り付き前にLvを把握できる。

## 9. 運用しながらの改善プロセス

### 9-1. 誤検知時の対処

ツールが誤検知した場合、以下の手順で辞書を改善する。

1. Discord通知のリンクから元記事を確認
2. 誤検知の原因を特定（例：「断固として反対」のような無関係文脈で「断固」が拾われた）
3. `config/keywords.yml` に除外パターンを追加、または該当キーワードをより具体的な表現に変更
   - 例：「断固」 → 「断固たる措置」「断固たる対応」
4. リポジトリにコミット → 次回実行から反映

### 9-2. 検知漏れ時の対処

検知漏れを発見した場合（後から「あの発言を拾えなかった」と気づいた場合）。

1. 発言の原文を確認
2. どのキーワードで拾えるべきだったかを特定
3. `config/keywords.yml` に新しいキーワードを追加
4. 過去ログ（`state/notified.json`）には影響しないが、今後同様の発言は拾えるようになる

### 9-3. 月次レビュー

月に1回、過去30日分の通知ログを見て以下を確認する。

- 誤検知率（通知された中で「これはノイズだった」と感じる割合）
- 検知漏れ件数（後から気づいた発言の数）
- 通知頻度（多すぎないか、少なすぎないか）

AIに「先月のFX警戒通知ログを評価して」と依頼すれば、ログ内容から改善提案を得られる。

### 9-4. ナレッジ自体の改善

ツールの精度改善とは別に、本ナレッジ自体も実運用で改善する。

- 過去事例（3-4）に新たな介入事例を追記
- Lv別の想定変動幅（3-2）を実測値で更新
- 監視対象人物（5章）を人事異動に応じて更新
